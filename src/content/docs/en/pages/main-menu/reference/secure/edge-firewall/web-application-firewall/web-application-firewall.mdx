**Web Application Firewall (WAF)** protects your applications from threats such as *SQL Injections*, *Remote File Inclusion (RFI)*, *Cross-Site Scripting (XSS)*, *Cross-Site Request Forgery (CSRF)*, *protocol violations* (such as empty body in POST), *Code Injection*, and other web vulnerabilities, including the [most common cyberthreats](https://www.azion.com/en/learning/websec/how-does-waf-protect-against-cyberthreats/#what-are-the-most-common-cyberthreats). WAF analyzes HTTP and HTTPS requests, detects, and blocks malicious activity before it reaches your application infrastructure.

WAF is a [Firewall](/en/documentation/secure/firewall/) module that operates in the seventh layer of the OSI model, the application layer, working as a barrier to filter and monitor traffic between your application and the requests from the internet. It's based on a request *scoring* methodology: each HTTP/HTTPS request is compared to a strict set of application standards and receives a *score* associated with a family of threats. According to the score, the request can be released or blocked directly in Azion's edge nodes, before the threat reaches its origin or causes any type of damage.

WAF can operate in two modes, **Learning** or **Blocking**, set in a [Rules Engine](/en/documentation/secure/firewall/reference/rules-engine/) rule with the **Set WAF Rule Set** behavior or via [Azion API](https://api.azion.com/#ace1b2f6-253a-48a1-bf58-4c5381e22c21). Read [WAF concepts](/en/documentation/secure/waf/concepts/) to understand the scoring model, the operation modes, and how WAF parses requests.

You can manage your WAF Rule Sets through [Azion Console](https://console.azion.com) or the [Azion API](https://api.azion.com/#ace1b2f6-253a-48a1-bf58-4c5381e22c21).

## Key features

### WAF Main Settings

The **Main Settings** section of a WAF Rule Set is configured by activating protection against specific threat families — SQL Injection, Remote File Inclusions (RFI), Directory Traversal, Cross-Site Scripting (XSS), File upload, Evading tricks, Unwanted Access, and Identified Attack — and setting a sensitivity level: Lowest, Low, Medium, High, and Highest. It's possible to enable and disable protection for each threat family individually through the **Active** switch.

See the [Threat types](/en/documentation/secure/waf/reference/rules-set/#threat-types) and [Sensitivity levels](/en/documentation/secure/waf/reference/rules-set/#sensitivity-levels) reference tables.

### WAF Tuning

**WAF Tuning** is an analytical tool that shows requests detected for each WAF rule, grouped by **rule ID**, with additional information such as **total hits, number of URIs, unique IPs**, and more, to help set up the **Allowed Rules** for the given WAF to match the best calibration for your application. You can filter by **Domain**, **Time Range**, **Network Lists**, **IP**, and **Countries**.

See [How to tune WAF](/en/documentation/secure/firewall/guides/tune-waf/).

### Allowed Rules and Exceptions

If internal traffic, tests, and false positives are being blocked by WAF, you can create *Allowed Rules* (referred to as **Exceptions** in the API) to explicitly allow legitimate request patterns. Each WAF rule has a unique numeric identifier, called `Rule ID`, which you can use to identify, filter, and allow/block specific behaviors of your application.

See the [WAF Exceptions reference](/en/documentation/secure/waf/reference/custom-allowed-rules/) for the fields, match zones, the full list of internal rules and their `Rule IDs`, and API management.

## Limits and plans

To configure a **WAF Rule Set**, which is what a WAF configuration is called, you must have a [Firewall](/en/documentation/secure/firewall/guides/firewall-configure-main-settings/) configuration with the **Web Application Firewall** module activated.

:::tip
**Increase limits** <br></br>
You can request to increase the limits based on your plan. Contact the [technical support team](/en/documentation/services/support/) to request it.
:::

These are the **default limits**:

| Scope | Limit |
| ----- | ----- |
| Debug Rules log size | 1.5 kB |
| WAF Tuning queries | 3 days |
| Request maximum payload size | 160 KB |

:::note
**Request Payload Size Limit**  
Payloads larger than 160KB will not be parsed by WAF and will trigger the rule 2 of WAF.
:::
